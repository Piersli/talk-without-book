# YouMind Skill: 读后无书 · 结构整理

> **完整云端版本**——所有结构整理动作（包括改 `道.md`、生成 Wiki）都在 YouMind 内完成。不依赖本地。

## skill 元数据

```yaml
name: 读后无书结构整理
display_name: 读后无书 · 结构整理
description: 跨书分析、候选道评审、道库更新、Wiki 生成、健康度审视——全部在 YouMind 内执行。读 board 里的材料，写回 board 里的材料/文档。
tags: [structure, dao-curation, cross-book, wiki, agentskills]
category: 知识 / 结构治理
visibility: private
```

## instructions（10 步）

1. **拉齐 board 里的所有材料**：用 `listFiles` 获取本 board 的 materials 和 documents。识别这几类：
   - 「道.md」material —— 当前道库（如果不存在，告诉用户先用 daily/extract 建立基础）
   - 「framework.md」material —— 判定规则
   - 所有形如「《{书名}》· 法」「《{书名}》· 术」「《{书名}》· 器」「《{书名}》· 势」「《{书名}》· meta」的 documents——按书分组
   - 已存在的 Wiki documents（如「Wiki · 道N · {标题}」）
   读取所有这些的内容。

2. **判断意图**（只选一个）：

   | 用户说 | 意图 |
   |---|---|
   | "审视道库"、"健康度"、"看看现状" | A. 道库健康度报告 |
   | "看候选道"、"有没有新道" | B. 候选新道评审 + 升格 |
   | "审视书 X" | C. 单书质量审视 |
   | "生成 Wiki"、"为道N 写一篇" | D. Wiki 生成 |
   | "跨书关联"、"找隐含联系" | E. 跨书隐含连接 |
   | "更新道库"、"把候选道并进来" | F. 道.md 实际更新 |
   | "全量重新整理"、"完整跑一遍" | G. 完整编译（A+B+E+D 串行）|

3. **执行分析**（按意图）：

   - **A 健康度**：道数、书数、各书节点数、孤立节点（缺上位道/法/术）、覆盖率（每条道有几本书触碰）、最近未触碰的道、节点质量分（法是否同义复述了道？）。
   - **B 候选道**：扫所有法节点，找上位道关联弱但跨多本书反复出现的主题。每个候选道写：临时标题、支撑法清单、与现有道的距离、升格/合并/暂留建议。
   - **C 单书审视**：法的因果机制是否具体？所有法是否有上位道？触碰强度分布？孤立节点？改进建议。
   - **D Wiki 生成**：围绕指定道，综合所有触碰它的书的视角，写一篇可独立阅读的 Wiki 文章。结构：「这条道在说什么」「各本书如何触碰」「核心法则」「可以做什么」「与其他道的关系」「相关主题」。
   - **E 跨书连接**：找多本书的法节点之间有隐含关联但没被显式标注的对（A 书的术是 B 书的法的实现 / A 和 B 描述同一现象的不同层面 / 等）。
   - **F 道.md 更新**：根据 B 的评审结果，准备具体的 `道.md` 修改方案——新增哪几条道、合并哪几条、修订哪几条的表述/检索问题/书源字段。
   - **G 完整**：依次跑 A、B、E、然后基于结果生成对应 Wiki。

4. **展示给用户 + 等确认**（对 B、F、G 必做；对 A、C、E 可跳到 step 5）：
   - 把分析结果以 markdown 形式展示给用户
   - 如果涉及对道.md 的修改（B 升格、F 更新），明确列出 diff：
     ```
     即将变更：
     - 新增：道{N+1}：{新道标题} —— 类别：{X}
     - 修订：道{M} 的"表述"字段（原文 vs 新文 并列）
     - 修订：道{K} 的"触碰此道的书籍"（追加 {书名}）
     ```
   - 问用户："以上变更要执行吗？"

5. **执行修改**（用户确认后）：
   - **改 道.md**：调用 `updateMaterial`（或如不支持原地更新，则 `createDocumentByMarkdown` 新版本 + 通知用户替换 material）。先在内存里拼好新版的完整 `道.md` markdown 文本，再整体 update。
   - **生成 Wiki**：每条道一个 `createDocumentByMarkdown`，命名「Wiki · 道N · {标题}」。如果该 Wiki document 已存在，覆盖（用户可在 YouMind 里看版本历史）。
   - **生成 health 报告**：用 `createDocumentByMarkdown` 创建「健康度报告 · {YYYY-MM-DD}」。
   - **生成 cross-book 连接**：用 `createDocumentByMarkdown` 创建「跨书隐含连接 · {YYYY-MM-DD}」。

6. **验证执行后的一致性**（替代本地的 validate.py）：
   - 重新读 `道.md` 确认变更已生效
   - 扫所有法节点的"上位道"字段——是否引用了不存在的道编号？
   - 道库的"触碰此道的书籍"字段——是否引用了 board 里没有对应 documents 的书？
   - 报告任何不一致

7. **报告执行结果**：

   ```
   ✓ 结构整理完成

   动作：{意图标签}
   修改：
   - 道.md：{修改概要}
   - 新增 Wiki：{N} 篇（链接列表）
   - 新增报告：{文档链接}

   验证：
   - 法 → 道引用：{正常/N 处异常}
   - 道 → 书引用：{正常/N 处异常}
   - 候选孤立节点：{N}

   下次浸泡（明天 8 点的 scheduled task）会自动用最新的道.md。
   ```

8. **可选：同步到本地**：如果用户问"我也想把这次更新同步到我本地的 talkwithbook"，告诉他们：
   > 在你本地装了 `talk-without-book` 的 Hermes/Claude Code 里说：
   > "从 YouMind 拉取 board『读后无书 Skill 工作台』的最新状态，同步到 $TWB_ROOT/"
   >
   > Hermes 会自动拉新版道.md、新书拆解、新 Wiki 写入本地。

9. **关键约束**：
   - **道是脊**——对道.md 的任何修改必须用户明确确认才执行。即使候选道在 5 本书里反复出现，也只是候选。
   - **不假装做了**——如果某个 YouMind API（如 updateMaterial）不可用，老实告诉用户"我只能 create 新版本，请你手动替换"。
   - **保留人类判断**——技能给材料、给观察、给提议；最终决策权在用户。

10. **小但重要的细节**：
    - 对 `道.md` 的修改必须保留原文件的整体格式（前置 "# 道 — 永恒不变的底层真相" 标题、横线分隔、字段顺序）
    - 每个被改动的道需要保留其历史 markdown 行号顺序，避免重排
    - 新增的道追加到文件末尾，不插在中间
    - 用户确认时，可以让用户用文字描述哪几条同意、哪几条不同意，不要强迫全部 yes/no

## 这个 skill 真正做到的事

| 旧的 YouMind structure（仅判断）| 新的（完整执行） |
|---|---|
| 生成提案文档 | 直接 `updateMaterial(道.md)` |
| 让用户去本地跑 compile.py | 自己生成所有 Wiki documents |
| 让用户去本地跑 validate.py | 自己用 LLM 校验一致性 |
| 用户必须装本地 Hermes 才能完成闭环 | YouMind 单独跑通完整闭环 |

## 与本地版的关系

- **本地版 twb-structure**（`scripts/compile.py` 等）：仍然存在，给装了 Claude/Hermes/OpenClaw 的用户用。对 1000 本书级别更高效（Python 并行处理）。
- **YouMind 版 读后无书结构整理**：给只用 YouMind 的用户用。对 1-50 本书级别完全够用。
- **两个版本可以并存**——本地是 source of truth，YouMind 通过 Hermes 同步保持一致。
