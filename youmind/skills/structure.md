# YouMind Skill: 读后无书 · 结构审视

> YouMind 上做**判断和提案**的 skill。不动文件、不跑脚本。本地的 Hermes/OpenClaw 才执行真正的改库动作。

## skill 元数据

```yaml
name: 读后无书结构审视
display_name: 读后无书 · 结构审视
description: 跨书分析、候选道评审、道库健康检查、Wiki 草稿生成。所有产出为提案文档，真正改库由本地执行器完成。
tags: [structure, dao-curation, cross-book, wiki-draft]
category: 知识 / 结构治理
visibility: private
```

## instructions（5 步）

1. **拉齐材料**：读取 board 里的所有 material：
   - 「道.md」（当前道库）
   - 所有「《{书名}》· 法.md」、「《{书名}》· 术.md」、「《{书名}》· 器.md」、「《{书名}》· 势.md」
   - 「framework.md」（判定规则）
   - 如果存在 Wiki 草稿 document，也读

2. **识别意图**：根据用户输入判断要做哪种结构动作（**只选一个**）：

   | 用户说 | 意图 |
   |---|---|
   | "审视道库"、"道库健康度"、"看看现在的道库情况" | A. 道库全景健康度报告 |
   | "看候选道"、"有没有新道"、"道索引该不该加" | B. 候选新道评审 |
   | "审视书 X" | C. 单书质量审视 |
   | "Wiki 草稿"、"为道N 写一篇 Wiki" | D. Wiki 草稿生成 |
   | "跨书关联"、"哪几本书在说同一件事" | E. 跨书隐含连接发现 |

3. **执行判断**（按意图执行对应分析）：

   - **A 健康度报告**：道数、书数、各书节点数、孤立节点（缺上位道/法）、覆盖率（每条道有几本书触碰）、最近未触碰的道。
   - **B 候选道**：扫所有法节点，找上位道关联弱（关联度 ≤1）但跨多本书反复出现的主题。每个候选道写：临时标题、支撑法清单（书+法编号）、与现有道的距离、升格 vs 合并 vs 暂留的建议。
   - **C 单书审视**：法是否都有上位道？因果机制是否具体（不是道的复述）？触碰强度分布？孤立节点？改进建议。
   - **D Wiki 草稿**：围绕指定道，综合所有触碰它的书的视角，写一篇可独立阅读的 Wiki 文章（按 `render.py` 的 system prompt 风格，但是输出为 document）。
   - **E 跨书连接**：找出多本书的法节点之间有隐含关联但没被显式标注的对。例如 A 书的术其实是 B 书的法的具体实现。

4. **生成提案文档**：调用 `createDocumentByMarkdown` 创建一份**提案**文档（不是最终改动），命名格式：「读后无书提案 · {YYYY-MM-DD} · {意图简称}」。文档结构：
   ```
   # 提案：{标题}

   ## 现状
   {基于 step 3 的分析}

   ## 建议
   {具体的改库动作清单——升格新道、合并、删除孤立节点、新增上位关系等}

   ## 怎么落地
   每一条建议都标注：可在 YouMind 直接做的（如复制文本到道.md material）vs 必须本地执行的（如跑 compile.py、改 dao/道.md 文件、生成 Wiki）。
   ```

5. **报告 + 移交本地**：给用户简短的总结，强调"这是提案，未改动任何东西"，并指引下一步：
   > 提案文档已生成在 board：{document URL}
   >
   > 如果同意提案里的改动，告诉本地 Hermes/Claude Code：
   > "执行 YouMind 上的提案文档 {url}，按里面写的改 $TWB_ROOT/dao/道.md 和相关文件"
   >
   > 本地 agent 会读这份提案，执行实际改库动作，并通过 `parse.py` + `validate.py` 验证。

## 关键约束

- **绝不假装修改了文件**。YouMind 没有写本地的能力，所有输出都是 document/proposal。
- **每条提案必须可执行**。不要含糊的"考虑合并"——具体到"把法27、法41 合并为一条新法 X，归属道5"。
- **道是脊**——道库的任何变更建议必须给出充分依据（在多本书里的支撑），而且明确标注"需要用户拍板"。
