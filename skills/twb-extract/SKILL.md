# 读后无书 · 内容拆解引擎

**读后无书（talk without book）** 三件套之一。

将一本书或一篇文章按 **道法术器势** 五维框架进行结构化提取，输出标准化节点文件——为之后的"无需翻书也能深度思考"奠基。

> 你和书的对话不是终点。终点是你能离开书，独自思考时，书里的道仍在场。
> 本 skill 帮你完成第一步：把书里的真东西提取下来，让它脱离原文也能存在。

## 输入

<input_path> #$ARGUMENTS </input_path>

## 前置准备

### 第一步：定位知识库根目录（`$TWB_ROOT`）

按以下顺序查找，**第一个匹配的就是 `$TWB_ROOT`**：

1. 环境变量 `$TWB_HOME`
2. `./talkwithbook/`（当前工作目录）
3. `~/talkwithbook/`
4. `~/Documents/talkwithbook/`
5. `~/Desktop/claude-code/talkwithbook/`（老用户兼容）

**如果都不存在**：询问用户是否在 `~/talkwithbook` 创建一个新知识库（这是产品默认的数据目录约定；用户也可以自定义）。同意则：
```bash
mkdir -p ~/talkwithbook/{dao,books,wiki}
touch ~/talkwithbook/dao/道.md
```
然后将 `$TWB_ROOT` 设为 `~/talkwithbook`。

> 注：「读后无书」是产品名，`talkwithbook` 是默认数据目录的传统命名（同字母缩写 twb，含义已从 talkwithbook 翻为 talk without book）。用户可以用任意目录名，只需设置 `$TWB_HOME` 或调整查找路径。

### 第二步：加载三个核心文件

1. **分级规范**：`Read ${SKILL_DIR}/framework.md`
   （SKILL_DIR 是本 skill 自己的目录——和 SKILL.md 在同一处）
   如果 skill 目录里没有 framework.md，回退去读 `$TWB_ROOT/framework.md`，再没有就告诉用户重新下载 skill。
2. **当前道索引**：`Read $TWB_ROOT/dao/道.md`（如果道索引为空，说明是第一本书——Phase 1 后会主动提议初始道集）
3. **输入文件**：`Read <input_path>`（确认文件存在并获取基本信息——文件大小、行数）

如果 `<input_path>` 为空，用 AskUserQuestion 询问文件路径。

确认输入文件后，向用户报告：
- 文件名、大小、行数
- `$TWB_ROOT` 是什么
- 预计的处理方式（是否需要分块）

用户确认后才开始。

## Phase 1：逐章阅读与概要

### 目标
理解全书结构，为后续提取提供上下文基础。

### 步骤

1. **判断文件大小并分块**
   - 如果 < 3000 行：一次性读取
   - 如果 3000-6000 行：分两块（前半/后半）
   - 如果 > 6000 行：分三块或更多，每块约 2000-2500 行

2. **启动阅读 Agent**
   - 对每个分块启动一个 Agent（并行）
   - Agent 的任务：对每一章输出
     - 章节名称
     - 核心论点（1-3句话）
     - 关键概念（列表）
     - 重要研究/案例/人物（列表）
     - 与当前每条道的关联度评分（0-3分）

   Agent Prompt 模板：
   ```
   请阅读以下书籍内容，对每一章提取：
   1. 章节名称
   2. 核心论点（1-3句话）
   3. 关键概念（列表）
   4. 重要研究/案例/人物（列表）
   5. 与以下每条道的关联度（0-3分）：
   {当前道索引，每条道一行}

   文件路径：{input_path}
   请读取第 {start_line} 到 {end_line} 行。
   用中文输出。这是纯阅读和分析任务，不需要写任何文件。
   ```

3. **汇总章节概要**
   - 收集所有 Agent 的输出
   - 合并为一份完整的全书章节概要
   - 识别全书的核心主题线索

4. **判断是否有候选新道**
   - 如果某个主题在多个章节中反复出现（覆盖 >30% 章节），但与现有道的关联度都 ≤1，标记为候选新道
   - 向用户报告候选道（如果有），但**不自动添加**，等 Phase 2 验证后再决定

## Phase 2：法的提取

### 目标
提取所有符合"法"定义的因果规律。这是质量要求最高的环节。

### 步骤

1. **启动法提取 Agent**

   Agent Prompt 要点：
   ```
   你是一个结构化思维专家。请为《{书名}》提取"法"维度的信息节点。

   ## 法的定义
   从道衍生出的可依赖的因果规律。每条法必须：
   1. 描述 A→B 的因果关系
   2. 有失效条件
   3. 能从某条道推导出来
   4. 不是道的同义复述（必须增加具体因果机制）

   ## 上位道索引
   {dao/道.md 完整内容}

   ## 法的节点格式
   ## 法N：[规律名称 — 一句因果表述]
   - **因果机制**：A 如何导致 B
   - **原文依据**：作者的论证和案例（章节+关键论据）
   - **适用边界**：在什么条件下成立，什么条件下失效
   - **上位道**：这条法从哪个道推导而来

   ## 全书概要
   {Phase 1 的章节概要}

   ## 要求
   - 穷举所有符合法定义的节点（预计30-40条）
   - 严格去重：与道同义的复述删除
   - 每条法必须增加道所没有的因果机制信息
   - 标注上位道
   - 如果发现现有道无法覆盖某些法，标注为"候选道N"
   - 末尾附一个道→法映射表

   请阅读原书文件 {input_path} 完成提取。
   输出为完整的 markdown 文件内容。
   ```

2. **审查法提取结果**
   - 检查节点数量是否合理（通常 25-45 条）
   - 检查是否每条法都有上位道
   - 检查道→法映射表是否覆盖了所有道
   - 向用户展示结果摘要，询问是否需要调整

## Phase 3：术/器/势的提取

### 目标
提取所有方法策略、工具框架、趋势判断。

### 步骤

1. **启动术器势提取 Agent**

   Agent Prompt 要点：
   ```
   你是一个书籍信息拆解专家。请从《{书名}》中提取"术"、"器"、"势"三个维度的信息节点。

   ## 维度定义
   "术"：具体的方法、策略、可操作的建议。回答"具体怎么做"。
   - 每条术标注上位法
   "器"：具体的工具、模板、案例、分类框架。回答"用什么"。
   - 每条器标注上位术
   "势"：条件、边界、趋势判断。回答"在什么条件下成立"。
   - 每条势标注时效性和影响范围

   ## 节点格式
   {参照 framework.md 中的术/器/势格式}

   ## 全书概要
   {Phase 1 的章节概要}

   ## 已提取的法索引
   {Phase 2 提取的法列表，每条法一行标题}

   请阅读原书文件 {input_path} 完成提取。
   术的"上位法"用描述性方式引用。
   ```

2. **审查结果**
   - 术是否都可直接行动（不可行动的降级为法）
   - 器是否都可直接复用
   - 势是否都标注了时效性

## Phase 4：质量检查与文件写入

### 质量检查清单

对照 framework.md 的质量检查清单逐项验证：

- [ ] 一本书触碰的道在 3-8 条之间
- [ ] 法没有与道重复（每条法增加了因果机制信息）
- [ ] 每条法标注了上位道
- [ ] 术都是可直接行动的
- [ ] 器都是可直接复用的
- [ ] 势标注了时效性
- [ ] 无跨维度重复

如有未通过项，标记给用户并建议修改。

### 文件写入

1. **创建书籍目录**
   ```bash
   mkdir -p "$TWB_ROOT/books/{Book_Name}"
   ```
   Book_Name 规则：英文标题用下划线连接（如 The_Social_Animal），中文标题直接用

2. **写入节点文件**
   - `$TWB_ROOT/books/{Book_Name}/法.md` — 法提取结果（Phase 2 输出）
   - `$TWB_ROOT/books/{Book_Name}/术.md` — 术提取结果
   - `$TWB_ROOT/books/{Book_Name}/器.md` — 器提取结果
   - `$TWB_ROOT/books/{Book_Name}/势.md` — 势提取结果

3. **生成 meta.md**
   ```markdown
   # 《{书名}》— 元信息

   > **书名**: {书名}
   > **作者**: {作者}
   > **出版年**: {出版年}
   > **核心主题**: {一句话总结}

   ## 本书触碰的道

   | 道 | 触碰角度 | 关联强度 |
   |---|---|---|
   {每条道一行，基于 Phase 1 的关联度评分}

   ## 节点统计

   | 维度 | 节点数 | 文件 |
   |------|--------|------|
   | 法 | N | 法.md |
   | 术 | N | 术.md |
   | 器 | N | 器.md |
   | 势 | N | 势.md |

   ## 与已有书籍的跨书连接
   {列出与已有书籍共享的道，以及各自的不同角度}
   ```

4. **更新全局道**
   - 更新 `$TWB_ROOT/dao/道.md` 中每条被触碰的道的"触碰此道的书籍"字段
   - 如果有已确认的新道（用户在 Phase 1/2 确认过），追加到 `$TWB_ROOT/dao/道.md`

5. **向用户报告**
   ```
   ✅ 《{书名}》拆解完成

   📁 文件位置：$TWB_ROOT/books/{Book_Name}/
   📊 节点统计：法 N | 术 N | 器 N | 势 N
   🔗 触碰的道：道1, 道3, 道4, 道7（列出编号）
   🆕 新增道：{如有}
   ```

6. **建议下一步**
   拆完之后**不**自动跑 Wiki 编译——那是 `twb:structure` 的职责。告诉用户：
   > 拆完了。要不要跑 `twb:structure` 把这本书并进知识库，重新生成 Wiki？
   
   或者用户也可以直接跑：
   ```bash
   python "$TWB_STRUCTURE_SKILL_DIR/scripts/compile.py" "$TWB_ROOT"
   ```
   
   **不要自己调 compile.py**——保持 skill 之间的边界清晰（拆书归 extract、整理归 structure）。

## 关键约束

1. **不要跳过 Phase 1**。没有全书概要直接提取法，质量会严重下降。
2. **法的提取必须用最强模型**。这是整个管线质量的瓶颈。
3. **候选新道必须人工确认**。不要自动往 dao/道.md 写入未确认的道。
4. **Agent 并行**。Phase 1 的多个阅读 Agent、Phase 2 和 Phase 3 尽可能并行。
5. **所有输出用中文**。节点内容、标题、描述全部中文。
6. **文件路径用绝对路径**。基于 `$TWB_ROOT`（前置准备里定位到的知识库根）。

## 对文章（非书籍）的处理

如果输入是一篇文章而非整本书：

- 跳过 Phase 1 的分块（文章通常 <3000 行）
- Phase 2 和 Phase 3 合并为一个 Agent 调用
- 文章的法通常只有 3-8 条，术/器/势各 2-5 条
- 存储路径：`books/{Article_Name}/`（和书籍相同结构）
